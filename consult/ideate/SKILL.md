# /ideate — 2フェーズひらめき生成

## 目的

**新しい発想・ひらめき**を、アンカリング（既存資料への寄り）を構造的に排除しつつ生成する。

設計上の工夫：
- **Phase 1（Wild Spark）は仕様書を物理的に見ない** → 発想が自由
- **Phase 2（Grounding）で仕様書と照合** → 発想と事実を接続
- 2フェーズを分けることで「発想の自由度」と「事実との整合性」を両立

---

## 使い方

### パターン1: 白紙から発散のみ

```
/ideate [テーマ]
```

→ Phase 1 のみ実行。飛躍した発想を5〜10個出力。  
→ 出力: `docs/[テーマ]_WILD.md`

### パターン2: 発散 + 仕様書との接地

```
/ideate [テーマ] --spec docs/[テーマ]_SPEC.md
```

→ Phase 1（仕様書ブラインド）→ Phase 2（仕様書と照合して3分類）。  
→ 出力: `docs/[テーマ]_WILD.md` + `docs/[テーマ]_INSIGHT.md`

---

## アーキテクチャ

```
テーマ（+ 任意で仕様書パス）
  │
  ├── Phase 1: Wild Spark（Opus）
  │   ├── 入力: テーマのみ（仕様書は絶対に渡さない）
  │   ├── 各アイデアに [根拠あり/推測/想像] ラベル強制
  │   ├── アイデア数: 5〜10個
  │   └── 出力: docs/[テーマ]_WILD.md
  │
  └── Phase 2: Grounding（Sonnet）※ --spec 指定時のみ
      ├── 入力: Phase 1 の発想 + 仕様書
      ├── 各アイデアの具体的主張を抽出
      ├── 3分類: 実現候補 / 発見候補 / 将来候補
      ├── ⚠️要確認マークを保持
      └── 出力: docs/[テーマ]_INSIGHT.md
```

---

## 実装：2フェーズの具体手順

### 共通変数

```bash
THEME="[テーマ]"
SPEC_PATH="[--spec 引数、省略可]"
SLUG=$(echo "$THEME" | tr ' ' '_' | tr -cd '[:alnum:]_-' | cut -c1-50)
WORK_DIR="/tmp/ideate_${SLUG}_$$"
mkdir -p "$WORK_DIR"
DOCS_DIR="docs"
mkdir -p "$DOCS_DIR"
```

### Phase 1: Wild Spark（Opus）

**絶対のルール**: プロンプトには**テーマ以外の情報を一切入れない**。
仕様書があっても、このフェーズでは読み込まない。

```bash
cat > "$WORK_DIR/p1_prompt.txt" <<'EOF'
あなたは発想の達人です。以下のテーマについて、
**既存の枠にとらわれない新しい発想**を5〜10個生成してください。

【テーマ】
THEME_PLACEHOLDER

【出力ルール】
1. アイデア数は5個以上10個以下
2. 各アイデアには以下の3ラベルのいずれかを必ず付ける:
   - [根拠あり]: 実在する事例・研究・統計で裏付けられる
   - [推測]: 論理的な推論・一般論に基づくが根拠は不確定
   - [想像]: 純粋な飛躍・創作。根拠はないが発想としての面白さで選んだ
3. [想像] ラベルも歓迎。発散フェーズなので飛躍は価値。
4. ただし [根拠あり] ラベルは**慎重に**。確証がないなら [推測] に落とす。
5. 「安全で実現容易な既存案のマイナーチェンジ」は採用しない。

【出力形式】
# [テーマ] — Wild Spark

## アイデア1: [タイトル] [ラベル]
- 概要: [3行程度]
- 着想源: [何から着想したか]
- 期待される効果: [起こりうる変化]
- リスク・前提: [これが成立するための条件]

## アイデア2: [タイトル] [ラベル]
...

## 発想のプロセスメモ
[どんな切り口で考えたか、何を避けたか]

【厳守】
- 各ラベルが5:3:2（根拠:推測:想像）程度のバランスが理想
- 全部が [根拠あり] になったら疑う（たぶん保守的すぎる）
- 全部が [想像] になったら現実との接点が弱い
EOF

sed -i "s|THEME_PLACEHOLDER|$THEME|g" "$WORK_DIR/p1_prompt.txt"

# Opusで実行（発想の核、ここはケチらない）
claude -p \
  --model opus \
  --no-session-persistence \
  --max-budget-usd 2.00 \
  < "$WORK_DIR/p1_prompt.txt" \
  > "$DOCS_DIR/${SLUG}_WILD.md" 2>&1

echo "Phase 1 完了: $DOCS_DIR/${SLUG}_WILD.md"
```

**PASS判定**：
- アイデア数が5未満 → 再実行
- [根拠あり] が全体の80%超 → プロンプトに「もっと飛躍せよ」を追加して再実行
- [想像] が90%超 → プロンプトに「現実との接点を1つは示せ」を追加して再実行

### Phase 2: Grounding（Sonnet、--spec 指定時のみ）

```bash
if [ -n "$SPEC_PATH" ] && [ -f "$SPEC_PATH" ]; then

cat > "$WORK_DIR/p2_prompt.txt" <<'EOF'
あなたはアイデア評価の専門家です。
以下の「Wild Spark発想」と「仕様書」を照らし合わせ、アイデアを3分類に振り分けてください。

【Wild Spark発想】
--- ここから発想 ---
WILD_OUTPUT_PLACEHOLDER
--- ここまで発想 ---

【仕様書】
--- ここから仕様書 ---
SPEC_CONTENT_PLACEHOLDER
--- ここまで仕様書 ---

【処理タスク】
1. Wild Spark の各アイデアから、具体的主張・数値・固有名詞を抽出する
2. それぞれを以下に分類：
   a. 仕様書に記載あり → 整合性◎
   b. 仕様書にはないが一般常識で確認可 → 整合性○
   c. 検証困難・仕様書と矛盾する → ⚠️要確認
3. その上で各アイデアを以下の3カテゴリに振り分け：
   - 【実現候補】 仕様書と整合し、すぐ検討可能
   - 【発見候補】 仕様書の盲点を突く、再評価の価値あり
   - 【将来候補】 実現困難だが長期的に価値がある

4. **アイデアを勝手に書き換えない**。評価のみ。
   ラベル [根拠あり/推測/想像] は Phase 1 のものをそのまま保持。

5. ⚠️要確認マークは**削除せず保持**する。

【出力形式】
# [テーマ] — Insight（接地済みひらめき）

## 【実現候補】仕様書と整合するアイデア

### アイデア名 [元ラベル]
- 元アイデア: [Phase 1 の概要]
- 仕様書との関連: [仕様書のどこと整合するか、該当節を引用]
- 実現のための次ステップ: [3個以内]
- ⚠️要確認: [検証困難な主張があれば]

## 【発見候補】仕様書の盲点を突くアイデア

### アイデア名 [元ラベル]
- 元アイデア: [Phase 1 の概要]
- 盲点の指摘: [仕様書に欠けている視点]
- 検証方法: [この発見を確認する手段]
- ⚠️要確認: [...]

## 【将来候補】実現困難だが価値あり

### アイデア名 [元ラベル]
- 元アイデア: [Phase 1 の概要]
- 実現困難の理由:
- それでも価値がある理由:
- ⚠️要確認: [...]

## 全体サマリー
- 実現候補: N個
- 発見候補: N個
- 将来候補: N個
- Phase 1 から削除されたアイデア: なし（全て保持）

【厳守】
- 発想の独創性を損ねない。評価者は判事ではなくキュレーター
- [想像] ラベルのアイデアも「将来候補」として拾い上げる
- ⚠️要確認マークは発見の手がかり。消さない
EOF

# 発想と仕様書を埋め込み
python3 -c "
with open('$WORK_DIR/p2_prompt.txt', 'r') as f: tmpl = f.read()
with open('$DOCS_DIR/${SLUG}_WILD.md', 'r') as f: wild = f.read()
with open('$SPEC_PATH', 'r') as f: spec = f.read()
out = tmpl.replace('WILD_OUTPUT_PLACEHOLDER', wild).replace('SPEC_CONTENT_PLACEHOLDER', spec)
with open('$WORK_DIR/p2_prompt.txt', 'w') as f: f.write(out)
"

# Sonnetで接地（評価タスク、Opus不要）
claude -p \
  --model sonnet \
  --no-session-persistence \
  --max-budget-usd 1.00 \
  < "$WORK_DIR/p2_prompt.txt" \
  > "$DOCS_DIR/${SLUG}_INSIGHT.md" 2>&1

echo "Phase 2 完了: $DOCS_DIR/${SLUG}_INSIGHT.md"

else
  echo "--spec 指定なし、Phase 2 スキップ"
  echo "発散アイデアは $DOCS_DIR/${SLUG}_WILD.md を確認してください"
fi
```

## モデル選択のデフォルト

| プロファイル | Phase 1 | Phase 2 | 用途 |
|---|---|---|---|
| `--standard`（デフォルト） | opus | sonnet | 通常の発想セッション |
| `--budget` | sonnet | sonnet | 軽量発想、量を多く出したい時 |
| `--pro` | opus | opus | 超重要案件（コスト注意） |

**Opus はデフォルトで Phase 1 に投入**。発想フェーズの質はスキル全体の価値を決めるため、ここだけはケチらない方針。

---

## アンカリング防止ルール（このスキルの核心）

1. **Phase 1 のプロンプトに仕様書文字列を一切含めない** — 物理的にブラインド
2. **Phase 1 用のサブClaude は `--no-session-persistence`** — 過去会話の影響を遮断
3. **Phase 2 で発想を書き換えない** — 評価のみ、改変禁止
4. **Phase 2 が「実現不能」と判定しても削除しない** — 将来候補として保持
5. **⚠️要確認マークは最終出力にも残す** — ユーザーの目視確認対象
6. **[想像] ラベルは価値として扱う** — 「飛躍=悪」ではない
7. **アイデア数は必ず5以上** — 少なすぎると発散とは言えない

---

## ハルシネーション防御（research との違い）

`/research` は「事実を磨く」スキル、`/ideate` は「発想を育てる」スキル。
両者でハルシネーションの扱い方が違う：

| 観点 | /research | /ideate |
|---|---|---|
| 目的 | 事実の精度 | 発想の質 |
| [推測] の扱い | ⚠️に隔離 | [想像] として歓迎 |
| ラベル軸 | 確実/高確率/推測 | 根拠あり/推測/想像 |
| チェック手段 | 独立ソースで裏取り | 仕様書との照合（Phase 2） |
| 最終判定者 | ユーザー（⚠️を目視） | ユーザー（3分類から選ぶ） |

**共通点**：どちらも**「確証の度合い」を明示**し、**ユーザーの最終判断に委ねる**設計。
Claude が勝手に「これは正しい」「これは良いアイデア」と断言しない。

## /research との連携ワークフロー

### 推奨：順次実行

```bash
# ステップ1: 事実ベースの仕様書を作る
/research [テーマ]
# → docs/[テーマ]_SPEC.md が生成される

# ステップ2: 仕様書を「見ないで」発散
/ideate [テーマ]
# → docs/[テーマ]_WILD.md が生成される（仕様書ブラインド）

# ステップ3: 発散結果と仕様書を照合
/ideate [テーマ] --spec docs/[テーマ]_SPEC.md
# → docs/[テーマ]_INSIGHT.md が生成される（Phase 2）
```

ステップ2と3を**必ず別コマンド**にしている理由：
- ステップ2でサブClaudeが仕様書を覗き見する余地を完全に排除
- 発散結果（WILD.md）だけを先に読めるので、かつき様が仕様書を見る前に発想を評価できる

### 並行実行も可

```bash
# /research と /ideate を独立に走らせる
/research ゲームデザイン改善 &
/ideate ゲームデザイン改善 &
wait
# 両方終わったら最終接地
/ideate ゲームデザイン改善 --spec docs/ゲームデザイン改善_SPEC.md
```

---

## ルール

1. **Phase 1 に仕様書を渡さない** — これは絶対。違反するとスキルの存在意義が消える
2. **Opus を Phase 1 から降格しない** — デフォルトで質を担保
3. **Phase 2 は評価のみ、発想の改変禁止**
4. **⚠️要確認は削除しない**
5. **アイデアを勝手に統合・削減しない** — 5〜10個を維持
6. **デフォルトはパターン1（発散のみ）** — 接地は明示的に `--spec` 指定時のみ
