# /consult — Web Claude 3ラウンド相談ループ

## コンセプト

「Claude が Claude に相談する」軽量マルチエージェントパターン。  
ハーネスが「実装→レビュー」を繰り返してコードを磨くように、  
consult は「質問→ファクトチェック→統合」を繰り返して**知識を磨く**。

1回の相談で終わると推測・ハルシネーションが混入しやすい。  
ラウンドを重ねることで「確実」と「推測」を分離し、  
信頼度の高い仕様書・設計書として出力する。

---

## アーキテクチャ

```
ユーザーの相談テーマ
  │
  ├── Round 1: 初稿生成
  │   ├── 役割・状況・質問をプロンプトに構造化
  │   ├── 不確かな部分は「要確認」と明記させる
  │   └── → 初稿テキスト
  │
  ├── Round 2: ファクトチェック
  │   ├── Round 1 で「要確認」になった箇所を抽出
  │   ├── 各項目に「確実／高確率で正しい／推測」を付けさせる
  │   └── → 検証済み訂正リスト
  │
  ├── Round 3: 統合・仕様書生成
  │   ├── Round 1 + Round 2 の確定情報を統合
  │   ├── 「推測」は ⚠️注意 セクションに隔離して明示
  │   └── → docs/ に .md ファイルとして保存
  │
  └── PASS 判定
      ├── 「推測」が残っていても許容範囲 → 完了
      └── 重大な「推測」が残る → Round 2 を追加で回す
```

## ラウンド構造（ハーネスのマイクロサイクル相当）

### Round 1: 初稿

**プロンプト構造:**

```
あなたは[専門家の役割]です。

【状況】
[プロジェクトの概要・技術スタック・現状]

【質問】
[具体的な問い。箇条書きで3〜5個]

不確かな部分は「要確認」と明記してください。
```

**判定基準:**
- 「要確認」が多い → Round 2 で集中的にファクトチェック
- 「要確認」が少ない → Round 2 でも重要箇所を深掘り

---

### Round 2: ファクトチェック

**プロンプト構造:**

```
前回の回答を受けて、以下の点を検証してください。

【検証依頼】
Q1: [Round 1 の「要確認」箇所1]
Q2: [Round 1 の「要確認」箇所2]
...

各回答に「確実」「高確率で正しい」「推測」のいずれかを付けてください。
```

**判定基準（ハーネスの PASS/FAIL 相当）:**

| 状態 | 判定 | 対応 |
|------|------|------|
| 全項目が「確実」または「高確率で正しい」 | PASS | Round 3 へ |
| 重要項目が「推測」のまま | FAIL | Round 2 を追加で実施 |
| 軽微な項目が「推測」のまま | 許容 | ⚠️注意として仕様書に明記して Round 3 へ |

---

### Round 3: 統合・仕様書生成

**プロンプト構造:**

```
Round 1・Round 2 の情報を統合して、最終仕様書を出力してください。

【確定情報】
- [Round 2 で「確実」になった事項]

【仕様書の構成】
# [テーマ名] 仕様書

## 1. 実装タスク一覧（チェックリスト形式）
## 2. [施策A]
## 3. [施策B]
...
## N. 実装優先度ロードマップ

ハルシネーション注意点は各施策の「⚠️注意」セクションに明記。
```

**出力先:** `docs/[テーマ名_大文字スネークケース].md`

## モデル選択ルール（ハーネスのエージェント使い分け相当）

| 用途 | モデル | 理由 |
|------|--------|------|
| 日常的な設計相談・戦略立案 | `claude-sonnet-4-6` | 安価・十分な品質 |
| 重要なアーキテクチャ判断 | `claude-opus-4-7` | 複雑な推論が必要な場合 |
| 高速な確認・単発の質問 | `claude-haiku-4-5-20251001` | コスト最小・簡単な検証 |

**Bashコマンドのテンプレート:**

```bash
claude -p --model claude-sonnet-4-6 "[プロンプト]" 2>&1
```

## ラウンド数の目安

| 相談の性質 | ラウンド数 | 目安工数 |
|-----------|-----------|---------|
| 日常的な調査・戦略立案 | 3回 | 5〜10分 |
| 重要な技術判断・設計 | 3〜4回 | 10〜20分 |
| 単純な確認・調査 | 2回 | 3〜5分 |
| 1回で十分な明快な質問 | 1回 | 1〜2分 |

## 成果物

| 成果物 | 場所 | 目的 |
|--------|------|------|
| 仕様書 | `docs/[テーマ].md` | 確定情報をチェックリスト形式で記録 |
| ラウンドログ | （保存不要・会話内で完結） | 中間過程は仕様書に統合して破棄 |

---

## ルール

1. **Round 1 は必ず「要確認」を含む形式で投げる** — 完璧な初稿を求めない
2. **「確実」と「推測」を必ず区別させる** — 混在したまま仕様書に入れない
3. **「推測」は仕様書の ⚠️注意 に隔離する** — 見えなくしない、削除もしない
4. **Round 3 は統合のみ** — 新しい質問を追加しない
5. **仕様書は `docs/` に保存する** — 会話が消えても知見が残るように
6. **モデルはデフォルト `claude-sonnet-4-6`** — 特別な理由がなければ変えない

## ハーネスとの対応表

| ハーネス | consult |
|---------|---------|
| マクロローテーション | 1回の相談セッション |
| マイクロサイクル | 1ラウンド |
| 実装→レビュー | 質問→ファクトチェック |
| `tsc --noEmit` | 「推測」が残っていないか確認 |
| PASS/FAIL 判定 | 「推測」が許容範囲か確認 |
| コードの品質向上 | 仕様書の信頼度向上 |
| `scripts/harness-check.js` | なし（出力は文書のため自動検証不要） |
