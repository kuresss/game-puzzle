# Harness Design

最終更新日: 2026-04-18  
対象: GAME1 Puzzle  
目的: 設計、実装、レビュー、再仕様化を反復し、ハルシネーション、仕様ズレ、バグ、未確認事項を継続的に減らす。

## 1. ハーネスの定義

このプロジェクトでいうハーネスは、次の一連の作業を固定手順として回す仕組みである。

1. 設計書を作る
2. 実装する
3. レビューする
4. 実装する
5. レビューする
6. 実装する
7. レビューする
8. 実装する
9. レビューする
10. 実装する
11. レビューする
12. 5 回分の実装・レビュー後にコード全体を再チェックする
13. 足りないものを書き起こす
14. 必要に応じて予備検索を行う
15. 仕様書を更新または再作成する
16. 次のローテーションに入る

この 1 から 16 までを 1 つの「マクロローテーション」と呼ぶ。マクロローテーションを 5 回繰り返す。

## 2. ローテーション構造

### 2.1 マイクロサイクル

マクロローテーション内の 1 回分の実装・レビューを、マイクロサイクルと呼ぶ。

```text
設計確認 -> 実装 -> 自動チェック -> 手動レビュー -> 記録
```

1 つのマクロローテーションには、マイクロサイクルを 5 回含める。

### 2.2 マクロローテーション

```text
仕様確認
  -> Cycle 1: 実装 / レビュー
  -> Cycle 2: 実装 / レビュー
  -> Cycle 3: 実装 / レビュー
  -> Cycle 4: 実装 / レビュー
  -> Cycle 5: 実装 / レビュー
  -> 全体コードチェック
  -> 不足事項の書き起こし
  -> 予備検索
  -> 仕様書更新
  -> 次ローテーションへ
```

## 3. 5 回ローテーション計画

| Macro | 主目的 | 完了条件 |
| --- | --- | --- |
| 1 | 中核ゲーム仕様の安定化 | 起動、移動、シャッフル、保存、仕様書整合が確認済み |
| 2 | UI/UX と操作品質の強化 | 操作可能/不可能が明確で、画面崩れがない |
| 3 | テストと自動チェックの拡張 | UI、保存、広告イベントのテスト方針が実装または記録済み |
| 4 | Android/広告/課金境界の検証 | AdMob/Billing の未実装範囲と実装済み範囲が分離されている |
| 5 | リリース前品質の棚卸し | 残リスク、ストア提出物、実機確認項目が再仕様化されている |

各マクロローテーションは、必ず 5 つのマイクロサイクルを含める。ただし、緊急バグが見つかった場合は、その修正を次のマイクロサイクルの主目的に繰り上げる。

## 4. 成果物

| 成果物 | 目的 |
| --- | --- |
| `docs/SPECIFICATION.md` | 現在の仕様と未実装範囲を定義する |
| `docs/QA_REVIEW.md` | 直近のレビュー結果と残リスクを記録する |
| `docs/HARNESS_DESIGN.md` | このハーネス設計を定義する |
| `docs/HARNESS_RUNBOOK.md` | 実行手順、コマンド、判定基準を定義する |
| `docs/templates/HARNESS_ROTATION_TEMPLATE.md` | 各ローテーションの記録テンプレート |
| `scripts/harness-check.js` | 無料で実行できる自動チェック |

## 5. レビュー観点

各レビューでは、以下を必ず確認する。

| 観点 | 確認内容 |
| --- | --- |
| 仕様整合 | 仕様書に書いたことが実装されているか |
| ハルシネーション | 外部事実、ゲームルール、SDK仕様を作り話で書いていないか |
| 操作性 | 実際に触って、遊び方が自然か |
| バグ | クリック、スワイプ、保存、再起動で破綻しないか |
| 表示 | 文字化け、白画面、レイアウト崩れがないか |
| テスト | 自動テストで中核ロジックが守られているか |
| Android | Debug APK が生成でき、エミュレーターで動くか |
| 残リスク | 未確認、未実装、実機依存の問題を隠していないか |

## 6. ハルシネーション防止ルール

1. 外部仕様、SDK、ストアポリシー、広告、課金、最新情報は予備検索で確認する。
2. 事実確認に使った情報は、仕様書またはレビュー記録に残す。
3. 推測は「推測」「未確認」「今後確認」と明記する。
4. 実装済みでないものを「完了」と書かない。
5. テストしていない環境を「確認済み」と書かない。
6. エミュレーター確認と実機確認を区別する。
7. 本番広告、本番課金、ストア審査は、実確認前に完了扱いしない。

## 7. 自動チェックゲート

各マイクロサイクル後、最低限以下を実行する。

```bash
npm run harness:check
```

依存関係の脆弱性確認が必要なサイクルでは、追加で以下を実行する。

```bash
npm run harness:audit
```

Android を触るサイクルでは、追加で以下を実行する。

```bash
npm run build
npx.cmd cap sync android
cd android
.\gradlew.bat assembleDebug
```

エミュレーター確認では、以下を最低限行う。

- 初回起動
- 合法タイル移動
- 不正タイル入力
- スワイプ
- シャッフル
- 再起動後復元
- logcat 致命エラー検索

## 8. 予備検索ゲート

以下に該当する場合は、必ず予備検索を行う。

- ゲームルール、歴史、既存ゲーム性を仕様に書く場合
- Capacitor、AdMob、Google Play Billing、Android SDK の仕様に触れる場合
- ストア公開、プライバシー、広告、課金、審査ポリシーに触れる場合
- 最新バージョン、推奨設定、非推奨APIを判断する場合
- 仕様書に外部事実を追加する場合

検索結果は `docs/QA_REVIEW.md` またはローテーション記録に残す。

## 9. 不足事項の書き起こし

各マクロローテーションの最後に、以下を必ず書き起こす。

| 分類 | 書く内容 |
| --- | --- |
| 未実装 | 実装されていない機能 |
| 未確認 | 実行していない確認 |
| 仕様ズレ | 仕様書と実装が一致しない箇所 |
| リスク | 本番公開前に問題になり得る箇所 |
| 次の実装候補 | 次ローテーションで取り組む順序 |

## 10. 完了判定

5 回のマクロローテーション完了時、以下を満たすことを目標にする。

- 仕様書が最新実装と一致している
- ハルシネーション確認の根拠が残っている
- 主要ゲームロジックに自動テストがある
- エミュレーターで主要操作が確認済み
- 未実装と未確認が明確に分離されている
- 残リスクが公開前チェックリストとして使える

## 11. 停止条件

以下の場合はローテーションを止め、先に問題を解消する。

- アプリが起動しない
- 白画面になる
- `npm test` が失敗する
- Debug APK が生成できない
- 仕様書に実装済みと書いた項目が実装されていない
- 外部根拠が取れない事実を断定している
- ユーザー確認が必要な判断が出た
